## 🔐 Система регистрации и авторизации

### Поддерживаемые методы входа

| Метод | Статус | Приоритет |
|-------|--------|-----------|
| Email + Пароль | ❌ Не реализовано | 🟠 Высокий |
| Telegram (через Telegram Login Widget или бот) | ❌ Не реализовано | 🟡 Средний |
| Google Sign-In | ❌ Не реализовано (только Email сейчас) | 🟡 Средний |
| Анонимный (технический) | ✅ Реализовано | — |

### Архитектура

**Текущий поток (Проблемный):**
```
Запуск → signInAnonymously() → пользователь логинится вручную в Настройках
```

**Новый поток:**
```
Запуск → Firebase проверяет кеш сессии
  ├─ Есть сохранённая сессия → автовход, показываем приложение
  └─ Нет сессии → показываем экран AuthScreen (регистрация/вход)
                    ├─ Войти через Email/Пароль
                    ├─ Войти через Telegram
                    └─ Продолжить без регистрации (ограниченный доступ)
```

### Экран `AuthScreen.kt`
✅ Сделано: Авторизация и профиль реализованы.

### Реализация входа через Telegram
- Пользователь нажимает "Войти через Telegram" → открывается Telegram-бот.
- Бот присылает одноразовый код подтверждения (OTP).
- Пользователь вводит код в приложении → Firebase Custom Token Auth.
- **Серверная часть:** небольшой Cloud Function (Firebase Functions) генерирует Custom Token по запросу от бота.

### Важные правила
- **Никогда не терять сессию:** Firebase хранит токен локально. При холодном старте — всегда проверять кеш перед показом AuthScreen.
- **Привязка аккаунтов:** Если анонимный пользователь нажал "Войти через Email" — мерджить анонимный аккаунт с реальным (не терять историю диагностик).
- **Восстановление пароля:** Ссылка "Забыл пароль" → Firebase `sendPasswordResetEmail`.

## 👤 Видение продукта: Ролевая модель пользователей

### Уровень 1 — Обычный пользователь (`user`)
Роль по умолчанию для всех скачавших приложение.
- ✅ Диагностика через ИИ-агента.
- ✅ Поиск решений **только в своей локальной базе** (глобальная база закрыта — **коммерческий задел**).
- ✅ Все найденные решения фоново синхронизируются с облаком, но **уходят в очередь премодерации** к экспертам (не публикуются напрямую в глобальную базу).
- ✅ **Офлайн-режим / Закладки (песочница):** Пользователь может заранее (дома, при наличии интернета) найти решения и сохранить их в «Закладки». Когда он приедет на место ремонта без интернета, он перебирает сохраненные варианты. Если одно из решений помогло — он отмечает его как «Мое решение». При первом подключении к интернету решение синхронизируется с облаком и уходит в очередь премодерации.
- ❌ Нет доступа к глобальной базе данных сообщества.
- ❌ Нет голосования (лайки/дизлайки) — это функция участника.

### Уровень 2 — Участник проекта (`contributor`)
Авторизованный пользователь, купивший подписку/доступ к глобальной базе.
- ✅ Все права обычного пользователя.
- ✅ **Доступ к глобальной базе данных** — проверенные решения, добавленные экспертами. Реальные кейсы, прошедшие строгую премодерацию.
- ✅ Голосование: лайки и дизлайки за решения в глобальной базе (формируют Карму автора).
- ✅ Ранние синхронизации и уведомления о новых решениях по «своим» маркам.

### Уровень 3 — Эксперт (`specialist`)
Проверенный специалист, получивший роль от Мастера или Администратора.
- ✅ Все права Участника.
- ✅ Решения эксперта **публикуются в глобальную базу сразу**, без премодерации.
- ✅ Доступ к **Экрану Премодерации**: видит все входящие решения от пользователей и участников.
  - Отображение **ИИ-оценки достоверности** (в %) — видна только экспертам и админам. Помогает понять, нужна ли перепроверка.
  - Может **одобрить**, **дополнить и одобрить**, или **отклонить** решение.
  - Может **удалять устаревшие или ошибочные** решения из глобальной базы.
- ✅ Без ограничений на количество ИИ-запросов.

### Уровень 4 — Администратор (`admin`)
Максимальный уровень. Зарезервирован за владельцами проекта.
- ✅ Все права Эксперта.
- ✅ Полная Панель управления: выдача/отзыв любых ролей по email.
- ✅ Публикация обновлений приложения (APK).
- ✅ Управление ключами API.
- **Текущие Админы:** `j.j.niccals2@gmail.com`, `slepa4ok@mail.ru` (жёстко прописаны в `SettingsViewModel.kt`).

## 🤖 Авто-парсер базы данных (RAG-архитектура)

**Цель:** Автоматически наполнять глобальную базу знаниями с форумов, чтобы снизить расход токенов и ускорить выдачу ответов.

**Логика работы (RAG flow в приложении):**
1. Запрос пользователя → **сначала ищет в Firebase (локальная база)**.
2. Нашёл → ответ мгновенно, **0 токенов потрачено**.
3. Не нашёл → запрос уходит в ИИ-агент (тратятся токены).

**Технический стек парсера (Python-скрипт, запускается отдельно):**
- **Источники:** Drive2.ru, MHH Auto, профильные форумы по автоэлектрике.
- **Отбор:** Только ветки с высоким числом просмотров/ответов.
- **Параллелизм:** 3 запроса одновременно (лимит ключа Dellmar API: `Параллельно: 3`). Реализация через `asyncio`.
- **API:** `https://api.dellmar.xyz/v1`, модель `claude-sonnet-4.6` или `gpt-5.6-sol`.
- **Промпт:** Жёстко структурированный JSON-вывод с полями: `carBrand`, `carModel`, `carYear`, `system`, `symptoms`, `errorCodes`, `solution`, `source: auto_parsed`.
- **Хранение:** Пуш в Firestore коллекцию `community_solutions`, статус `pending` (требует одобрения экспертом).
- **Нормализация:** Приведение всех марок/моделей к единому стандарту (нет дублей «Шкода»/«Skoda»).

## 🐛 Критические Баги (Must Fix)

### Баг #1 — Решения добавляются без автора 🔴 КРИТИЧЕСКИЙ
**Описание:** Пользователь не пользовался приложением весь день, вечером открыл, выполнил поиск, добавил решение — оно сохранилось **без автора**, хотя Профиль показывает авторизацию.
**Причина:** Сессия Firebase Auth не восстановилась при холодном старте. `ViewModel` использует закешированный пустой `email`.
**Решение:** Перед записью решения в БД всегда брать актуальные данные из `FirebaseAuth.getInstance().currentUser`, а не из кеша `ViewModel`.

### Баг #2 — Постоянное окно «Доступна новая версия» ✅ Исправлен (v1.7.8)
**Причина:** `versionCode` в `build.gradle.kts` не совпадал с опубликованным значением в Firebase.
**Правило:** При каждом релизе синхронно поднимать `versionCode` в коде И в Админке Firebase.

### Баг #3 — «Object does not exist» при публикации APK ✅ Исправлен (v1.7.8)
**Причина:** `UploadTask.await()` отдавал управление до завершения загрузки файла.
**Решение:** Заменён на `suspendCancellableCoroutine` с `addOnSuccessListener`.
